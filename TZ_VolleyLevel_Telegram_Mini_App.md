### Техническое задание (ТЗ)

**Проект:** VolleyLevel — геймификация школы волейбола  
**Формат:** Telegram Bot + Telegram Mini App (WebApp) + Backend API  
**Репозиторий:** `Gamification_Volley_v2/`  
**Язык интерфейса:** RU

---

### 1. Введение

#### 1.1. Цель продукта
- Повысить вовлечённость учеников через **уровни, XP, навыки, достижения, рейтинги**.
- Упростить фиксацию тренировок:
  - тренер записывает тренировку ученику;
  - ученик отмечается через **QR** (в зале).
- Дать администрации инструменты управления пользователями, настройками, контентом (ачивки/навыки/локации/QR).
- (Опционально) давать персональные советы “AI‑тренера” на основе прогресса навыков.

#### 1.2. Общая архитектура
- **Telegram Bot**: точка входа, запуск Mini App (создание и настройка через BotFather).
- **Telegram Mini App (Frontend)**: интерфейс ученика/тренера/админа, работает внутри Telegram WebView.
- **Backend API (Express + SQLite/sql.js)**: хранение данных, бизнес‑логика, роли/права, валидация Telegram `initData`.

---

### 2. Термины и определения
- **XP**: опыт пользователя.
  - `xp` — текущий опыт в рамках текущего уровня.
  - `totalXp` — суммарный накопленный опыт за всё время.
- **Уровень**: вычисляется на основе XP и настроек системы.
- **Навык**: отдельная ветка прогресса (пример: подача/приём/атака/блок/пас/физо). Навыки можно добавлять/отключать.
- **Тренировка**: событие, увеличивающее `trainingsCompleted` и влияющее на streak.
- **Streak (стрик)**: серия дней с тренировками.
- **QR‑код**: сущность, которую ученик сканирует 1 раз в сутки (лимит); может быть пресетом тренировки или “legacy” XP.

### 4. Авторизация и безопасность (Telegram)

#### 4.1. Требования безопасности
- В production режиме **все** запросы к `/api/*` должны содержать валидный Telegram `initData`.
- Валидация `initData` выполняется на сервере по HMAC‑SHA256 с использованием `BOT_TOKEN`.
- При отсутствии `initData` в production — вернуть **401** с сообщением “Откройте приложение через Telegram”.

#### 4.2. Dev‑режим
- В dev‑режиме (когда `NODE_ENV != production`) сервер допускает mock‑пользователя при отсутствии `initData` для локальной разработки.

### 5. Регистрация и жизненный цикл пользователя

#### 5.1. Идентификация
- Источник идентификатора: Telegram user id (`telegram_id`).

#### 5.2. Создание пользователя
- При первом заходе Mini App сервер должен:
  - создать запись пользователя с `role=STUDENT`;
  - заполнить имя/аватар из Telegram данных (fallback: “Player”);
  - присвоить значения города/филиала по умолчанию (если не переданы);
  - инициализировать навыки (все включённые навыки с базовым значением).

### 6. Функциональные требования (Mini App)

#### 6.1. Общая навигация
- Нижнее меню:
  - **Главная** — для всех ролей.
  - **Навыки** — только для `STUDENT`.
  - **Ученики (тренерский экран)** — для `TRAINER`/`ADMIN`.
  - **Управление (админ‑панель)** — только для `ADMIN`.
  - **Рейтинг** — для всех ролей.

#### 6.2. Экран “Главная” (Dashboard)
- Отображать карточку пользователя: имя, аватар, город/филиал, уровень, титул.
- Отображать прогресс уровня: `xp / xpNext` и прогресс‑бар.
- Отображать статистику: `totalXp`, `trainingsCompleted`, `streak` (если > 0).
- Для `STUDENT`:
  - кнопка “Отметиться в зале” открывает QR‑сканер;
  - кнопка недоступна, если пользователь уже сканировал QR сегодня (состояние “уже отмечено сегодня”).

#### 6.3. QR‑сканер
- Запросить доступ к камере.
- Сканировать QR и извлечь `qrId`:
  - поддержать ввод `qrId` напрямую (для теста/фолбэка);
  - (желательно) поддержать извлечение `qrId` из URL формата `/scan/<qrId>`.
- После успешного сканирования:
  - начислить XP пользователю;
  - увеличить количество тренировок;
  - обновить streak;
  - показать итог (например: “+X XP за <название QR>”);
  - при наличии новых достижений показать toast‑уведомление.

#### 6.4. Экран “Профиль ученика”
Доступен:
- как профиль текущего пользователя (в т.ч. редактируемый);
- как просмотр профиля другого пользователя (из рейтинга/списка учеников).

Функции:
- Прогресс уровня.
- Диаграмма навыков (радар).
- Карточки навыков: уровень 1–10 + прогресс к следующему уровню навыка.
- Достижения:
  - список всех достижений;
  - состояние “получено/не получено”;
  - просмотр условий достижения в модальном окне.
- История активности:
  - элементы с типом: `qr`, `training/preset`, `xp_bonus`, `xp_deduction`;
  - отображать: дату, название/фокус, полученный XP.

Редактирование собственного профиля:
- имя;
- аватар (галерея пресетов);
- город/филиал.

#### 6.5. Экран “Навыки” (для STUDENT)
- Радар‑диаграмма.
- Список навыков с:
  - уровнем навыка (1–10);
  - титулом уровня навыка;
  - XP по навыку;
  - прогресс‑баром.
- Блок “Советы AI‑тренера”:
  - генерирует 3 коротких совета на русском (с волейбольным сленгом);
  - кнопка “Обновить советы”.
- Если AI‑ключ не задан, показывать fallback‑совет (без падения UI).

#### 6.6. Экран “Рейтинг”
- Режимы:
  - **Игроки** — сортировка по `totalXp` (только `STUDENT`, только `trainingsCompleted > 0`);
  - **Филиалы** — агрегирование по (город, филиал);
  - **Города** — агрегирование по городу.
- Переход в профиль игрока по клику на строку.

#### 6.7. Экран “Тренер” (TRAINER/ADMIN)
Общее:
- выбор ученика.

Режимы:
- **Тренировка (пресеты)**:
  - кнопки‑пресеты (например “Атака + Блок”, “Прием + Пас” и т.п.);
  - запись тренировки ученику, начисление XP на соответствующие навыки.
- **Несколько навыков**:
  - тренер вручную задаёт набор (skillId, xpAmount);
  - запись тренировки ученику.
- **Бонус XP**:
  - начисление XP (опционально в конкретный навык);
  - **не считается тренировкой** и не увеличивает `trainingsCompleted`.

#### 6.8. Админ‑панель (ADMIN)
Разделы:
- **Статистика**: общее число учеников, общий XP сети, рейтинг филиалов.
- **Ученики**:
  - фильтры: город, филиал, поиск, диапазоны уровней/тренировок/streak/XP, роль;
  - операции:
    - редактирование профиля;
    - редактирование статистики (xp/totalXp/level/trainings/streak);
    - изменение роли;
    - начислить/списать XP (опционально по навыку);
    - выдать/отозвать достижения.
- **QR**:
  - генерация QR для тренировок (пресеты) + кастомные параметры:
    - город/филиал;
    - `maxUses` (лимит сканирований);
    - `expiresAt` (срок действия, например “в часах”);
    - привязка достижения (опционально);
  - список QR:
    - показать QR‑картинку;
    - скачать/распечатать;
    - удалить.
- **Награды (Достижения)**:
  - создание/редактирование/удаление;
  - условия: `minLevel`, `minTrainings`, `minStreak`, `minTotalXp`, `minSkillValue`.
- **Локации**:
  - добавление/удаление городов;
  - добавление/удаление филиалов.
- **Навыки**:
  - добавление навыка;
  - включение/отключение навыка.
- **XP**:
  - настройка `xpPerLevel`, `multiplier`, `weekendBonus`.

### 7. Бизнес‑правила и расчёты

#### 7.1. Расчёт XP до следующего уровня
Формула:
- `xpNext = floor(xpPerLevel * multiplier^(level-1))`

#### 7.2. Правила начисления XP
- Начисление XP обновляет:
  - `xp`, `totalXp`, `level` (при необходимости).
- В выходные дни применяется множитель (`weekendBonus`, по умолчанию x2) к начислениям:
  - тренировки;
  - QR‑сканирование;
  - бонус XP.

#### 7.3. Правила “тренировка” vs “бонус XP”
- Запись тренировки:
  - увеличивает `trainingsCompleted` на 1;
  - обновляет `streak`.
- Бонус XP:
  - **не** увеличивает `trainingsCompleted`;
  - сохраняется в истории как отдельный тип события.

#### 7.4. QR‑лимиты
- Пользователь может выполнить **не более 1 QR‑сканирования в сутки**.
- QR может иметь:
  - `expiresAt` (запрет начисления после истечения);
  - `maxUses` (запрет после достижения лимита).

#### 7.5. Рейтинги
- В рейтинги попадают только `STUDENT` с `trainingsCompleted > 0`.

### 8. Достижения (Achievements)

#### 8.1. Модель достижения
Поля:
- `id`, `title`, `description`, `imageUrl`, `conditions`.

Условия:
- `minLevel`
- `minTrainings`
- `minStreak`
- `minTotalXp`
- `minSkillValue: { skill, value }`

#### 8.2. Разблокировка
- Автоматическая проверка достижений выполняется после:
  - тренировки;
  - QR‑сканирования;
  - начисления XP.
- Достижение может быть привязано к QR и выдаваться при сканировании QR.
- Админ может выдать/отозвать достижение вручную.

### 9. QR‑коды

#### 9.1. Модель QR
Поля:
- `id`, `title`, `city`, `branch`, `xpAmount`
- опционально: `skillId` (legacy), `skills[]` (preset), `achievementId`, `expiresAt`, `maxUses`, `usesCount`, `isTrainingPreset`.

#### 9.2. Статистика QR
- Для QR должна быть доступна статистика:
  - `totalScans`, `uniqueUsers`, `totalXpAwarded`.

### 10. API (контракт)

#### 10.1. Общие требования
- Базовый префикс: `/api`
- Заголовок авторизации: `X-Telegram-Init-Data: <initData>`
- Коды ошибок:
  - 401 — не авторизован (нет/невалидный initData)
  - 403 — недостаточно прав
  - 400 — ошибка входных данных
  - 500 — внутренняя ошибка

#### 10.2. Основные эндпоинты (перечень)
- Users:
  - `GET /users/me`

  - `GET /users/:id`
  - `POST /users/award-xp` (TRAINER/ADMIN)
  - `POST /users/deduct-xp` (TRAINER/ADMIN)
  - `POST /users/log-training` (TRAINER/ADMIN)
  - `POST /users/scan-qr`
  - `PUT /users/:id/role` (ADMIN)
  - `PUT /users/:id/profile` (self или ADMIN)
  - `PUT /users/:id/stats` (ADMIN)
  - `POST /users/:id/achievements/:achievementId` (ADMIN)
  - `DELETE /users/:id/achievements/:achievementId` (ADMIN)
  - `POST /users/:id/recalculate-skills` (ADMIN)
- Rankings:
  - `GET /rankings/players`
  - `GET /rankings/branches`
  - `GET /rankings/cities`
  - `GET /rankings/skills/:skillId`
  - `GET /rankings/streaks`
- Achievements:
  - `GET /achievements`
  - `POST /achievements`
  - `PUT /achievements/:id`
  - `DELETE /achievements/:id`
- QR Codes:
  - `GET /qrcodes`
  - `GET /qrcodes/:id`
  - `POST /qrcodes` (TRAINER/ADMIN)

- Skills:

  - `GET /locations`
  - `GET /locations/cities`

  - `GET /settings`

### 12. Конфигурация и окружение

#### 12.1. Переменные окружения
- `BOT_TOKEN` — токен Telegram‑бота (обязательно для production).
- `NODE_ENV` — режим работы (`production` включает строгую авторизацию).
- `PORT` — порт API (по умолчанию 8000).
- `DB_PATH` — путь до файла БД (опционально).
- `DEEPSEEK_API_KEY` / `VITE_DEEPSEEK_API_KEY` — ключ для AI‑советов (опционально).

#### 12.2. Деплой (общее)
- Frontend отдаётся как статические файлы (например `dist/`) через nginx.
- Backend проксируется как `/api/*` на Node/Express.
- HTTPS обязательно для нормальной работы Telegram WebApp.

### 13. Критерии приёмки (Acceptance Criteria)

#### 13.1. Авторизация
- В production:
  - без валидного Telegram `initData` API возвращает 401;
  - с валидным `initData` пользователь создаётся/получается корректно.

#### 13.2. Ученик
- Видит профиль/навыки/ачивки/историю.
- Может сканировать QR:
  - 1 раз в сутки успешно;
  - повторно в тот же день получает корректную ошибку.
- После первой тренировки появляется в рейтинге игроков.

#### 13.3. Тренер
- Может записать тренировку ученику:
  - пресет;
  - кастомный набор навыков.
- Может начислить бонус XP (и это **не** увеличивает `trainingsCompleted`).

#### 13.4. Админ
- Может управлять ролями.
- Может редактировать профиль/статистику ученика.
- Может создать/удалить QR, задать лимиты/срок действия.
- Может создать/редактировать/удалить достижения и выдавать/отзывать их ученикам.
- Может управлять локациями и навыками.

---

### 14. Что не входит в объём (опционально уточнить)
- Полноценная чат‑логика “бота” (команды, диалоги) — продукт реализован как Mini App.
- Платёжные функции, подписки, интеграции CRM.
- Push‑уведомления вне Telegram (email/SMS).
